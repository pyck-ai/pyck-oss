# Contributing to pyck

Contributions to pyck are welcome! Follow these guidelines to ensure smooth collaboration.

## Commit and Branch Standards

Please see [docs/COMMIT_STANDARDS.md](./docs/COMMIT_STANDARDS.md) for detailed information on:
- Commit message format (Tim Pope style)
- Branch naming conventions
- Examples and best practices

## Workflow

1. **Fork the repository**
2. **Create a feature branch** following the naming convention
3. **Make your changes** following code quality standards
4. **Add tests** for new functionality (aim for >80% coverage)
5. **Run validation**:
   ```sh
   task test   # All tests must pass
   task lint   # No linting errors allowed
   task build  # Build must succeed
   ```
6. **Commit** with clear messages following the format above
7. **Open a pull request**:
   - Title should reference the issue
   - Body must describe:
     - What changed
     - Why it was necessary
     - How to test it
   - Include "Fixes: #issue" in PR body or commit message
   - Ensure all CI checks pass (lint, test, build, version check)


## Code Quality Standards

### Go Code

- Follow Go idioms and effective Go patterns
- Handle errors explicitly (no panic in production code)
- Add tests for new features (aim for >80% coverage)
- Document all exported functions, types, and packages
- Use contexts for cancellation and timeouts
- Prefer composition over inheritance
- Use dependency injection for testability

### GraphQL

- Use directives for Federation (`@key`, `@external`, `@requires`)
- Implement Relay-style pagination for lists
- Add ordering fields via `entgql.OrderField`
- Use mutations for write operations only

### Temporal Workflows

- Workflows MUST be deterministic (no randomness, no wall-clock time)
- Workflow structs MUST NOT contain state (instantiated once per worker)
- Activity structs CAN contain shared state (DB connections, caches)
- Use `workflowgen` for automatic workflow registration
- Always set activity timeouts

### GitHub Actions JavaScript
- Avoid deeply nested if-else chains
- Extract complex logic into functions
- Use descriptive variable names
- Prefer early returns over nested conditions

## Code Review Focus

When reviewing code, focus on:

**ALWAYS review:**
- `ent/schema/*` - Schema definitions (handwritten data models)
- `ent/migrate/migration/*` - Migration files (database changes)
- All other Go source files
- Configuration files
- Documentation

**IGNORE (autogenerated):**
- All other `ent/*` files (except schema and migrations)
- Files containing "// Code generated" header
- Vendor directories
- Built artifacts

## Security Guidelines

- Never commit secrets, tokens, or credentials
- Always validate user input at GraphQL resolver level
- Follow principle of least privilege
- Use Zitadel for authentication (JWT tokens)
- Enforce tenant isolation in all queries
- SQL injection prevented by Ent parameterized queries
- Report security concerns to maintainers immediately

## Testing Requirements

Before committing code, ensure all validation passes:

```sh
task test   # All tests must pass
task lint   # No linting errors allowed
task build  # Build must succeed
```

Aim for >80% test coverage on new features.

## Common Pitfalls

Watch out for these common mistakes:

- **Database**: Missing `tenant_id` in queries, forgetting `NotDeletedFilter()` for soft deletes
- **Workflows**: Non-deterministic code, missing activity timeouts, storing state in workflow structs
- **Code Generation**: Editing generated files, not running `task generate` after schema changes
